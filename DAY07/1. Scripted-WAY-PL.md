# Jenkins Scripted Pipeline In-Depth Documentation

## 1. Introduction

In Jenkins, at the beginning many teams start with **Freestyle Jobs** because they are easy to create from the UI.
But when the project grows, Freestyle jobs create many problems.

That is why most real-time companies move to **Jenkins Pipelines**.

Pipelines help us automate the full software delivery process in a proper, repeatable, and manageable way.

A Jenkins Pipeline means:

Code checkout
Build
Test
Code quality check
Artifact creation
Artifact upload
Deployment
Validation
Notifications

All these steps are written as code.

This is called **Pipeline as Code**.

---

# 2. What is a Freestyle Job?

A Freestyle Job is the traditional Jenkins job where we configure everything manually from the Jenkins UI.

Example:

Source code management in one section
Build trigger in one section
Build steps in one section
Post-build action in another section

This works for small tasks, but in real projects it becomes difficult.

---

# 3. Problems with Freestyle Jobs

## 3.1 Manual configuration problem

In Freestyle jobs, most settings are created manually in the Jenkins UI.

Problems:
If Jenkins server crashes, job configuration may be lost
Difficult to recreate the same job in another Jenkins server
Manual mistakes happen often
Same configuration has to be repeated for multiple jobs

Real-time issue:
One team created 25 freestyle jobs manually. After Jenkins migration, recreating all jobs took many hours and some settings were missed.

---

## 3.2 No version control for job configuration

Freestyle job configuration is stored inside Jenkins, not properly inside Git like application code.

Problems:
No proper version tracking
Who changed what is not easy to track
Rollback is difficult
No peer review for job changes

Real-time issue:
One engineer changed the build command in UI from `mvn clean package` to `mvn package -DskipTests`. Later production issue came, but nobody knew when that setting changed.

---

## 3.3 Hard to manage complex CI/CD flow

Freestyle jobs are okay for a simple build.
But real projects need multiple steps:

Checkout
Compile
Unit test
Static code scan
Package
Upload to Nexus
Deploy to Dev
Integration testing
Approval
Deploy to QA
Deploy to Prod

Managing this in one Freestyle job becomes messy.

Sometimes teams create multiple freestyle jobs and chain them together.
That also becomes difficult to maintain.

---

## 3.4 No proper reusability

If 10 projects need the same process, in Freestyle jobs you must configure 10 jobs manually.

Problems:
Duplicate work
Inconsistent settings
One job may have Sonar step, another may miss it
One job may deploy wrong artifact path

---

## 3.5 Difficult troubleshooting

In Freestyle jobs, logic is scattered in UI fields.

Problems:
Hard to understand full flow
Hard to debug
Hard for new team members to understand
Logic is not centralized

---

## 3.6 No easy branching logic

In real projects we need logic like:

If branch is dev, deploy to dev
If branch is release, deploy to QA
If branch is main, deploy to prod

Freestyle jobs cannot handle such logic cleanly.

---

## 3.7 No proper parallel execution

Sometimes we want:

Run unit tests in parallel
Run lint and security scan in parallel
Build multiple microservices in parallel

Freestyle jobs are not good for advanced parallel workflow.

---

## 3.8 Promotion flow becomes difficult

Real pipeline requirement:

Build once
Promote same artifact to Dev
Promote same artifact to QA
Promote same artifact to UAT
Promote same artifact to Prod

Freestyle job setup often rebuilds artifact again and again, which is bad practice.

---

## 3.9 Hard to audit and review

Freestyle jobs are UI-driven.
Pipeline code can be reviewed like application code.

With Freestyle:
No pull request review
No easy standardization
No clean audit history

---

# 4. Why We Move to Jenkins Pipelines

We move from Freestyle jobs to Pipelines because Pipelines solve most real-time problems.

## Main reasons

Everything is written as code
Can store in Git
Can review changes
Can reuse code
Can add conditions
Can run stages clearly
Can integrate better with modern DevOps tools
Can build CI/CD in a structured way

This is why companies prefer pipelines.

# 5. What is Jenkins Pipeline?

A Jenkins Pipeline is a set of steps written in code that defines the build, test, and deployment process.

Instead of clicking options in UI, we write a `Jenkinsfile`.

That file stays in the source code repository.

Whenever Jenkins runs the job, it reads the pipeline steps from that file.

This is called **Pipeline as Code**.

---

# 6. Types of Jenkins Pipelines

There are mainly two types:

## 6.1 Scripted Pipeline

Written in Groovy style
More flexible
Good for advanced logic
More coding required

## 6.2 Declarative Pipeline

More structured
Easy to read
Easy for standard CI/CD
Modern teams often prefer this for simple-to-medium flows

Your request is mainly about **Scripted Pipeline**, so let us go deep into that.

# 7. What is Scripted Pipeline?

Scripted Pipeline is a Groovy-based Jenkins pipeline where we write logic using programming style.

It gives full control to the DevOps engineer.

Example structure:

```groovy
node {
    stage('Checkout') {
        git 'https://github.com/example/repo.git'
    }

    stage('Build') {
        sh 'mvn clean package'
    }

    stage('Test') {
        sh 'mvn test'
    }
}
```

Here, we are manually writing the execution flow.

---

# 8. Why Scripted Pipeline is Powerful

Scripted pipeline is powerful because it supports:

Custom logic
If-else conditions
Loops
try-catch
Functions
Parallel blocks
Dynamic stage creation
Advanced handling for real-time scenarios

This is why experienced Jenkins engineers still use Scripted Pipeline for complex use cases.

# 9. Core Building Blocks of Scripted Pipeline

## 9.1 node

`node` means execute the pipeline on a Jenkins agent or worker.

Example:

```groovy
node {
    echo "Running on Jenkins node"
}
```

Why needed:
Jenkins needs a workspace and executor to run the job.

---

## 9.2 stage

A stage divides pipeline into visible logical sections.

Example:

```groovy
stage('Build') {
    sh 'mvn clean package'
}
```

Why needed:
Easy visibility in Jenkins UI
Easy debugging
Easy tracking of failure point

---

## 9.3 steps inside stage

Inside each stage we run commands.

Examples:
`sh` for Linux shell
`bat` for Windows
`git` for checkout
`echo` for logs
`tool` for tool path
`withCredentials` for secrets

---

# 10. Basic Scripted Pipeline Example

```groovy
node {
    stage('Checkout') {
        git branch: 'main', url: 'https://github.com/example/app.git'
    }

    stage('Build') {
        sh 'mvn clean package'
    }

    stage('Test') {
        sh 'mvn test'
    }

    stage('Archive') {
        archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
    }
}
```

Explanation:

Checkout stage pulls source code
Build stage creates artifact
Test stage runs tests
Archive stage stores artifact in Jenkins

# 11. Real-Time Scripted Pipeline for Java Application

```groovy
node {

```

# 12. Why Scripted Pipeline is Better Than Freestyle in Real Projects

## 12.1 Full flow visible in one file

In Freestyle:
Build steps in one tab
Git in another tab
Post-build in another tab

In Pipeline:
Everything is in one file

That means:
Easy to read
Easy to share
Easy to review
Easy to maintain

---

## 12.2 Git-based version control

Jenkinsfile is stored in Git.

Benefits:
Track every change
Pull request review possible
Rollback possible
Environment consistency

---

## 12.3 Reusability

We can create common methods and reuse them.

Example:

```groovy
def buildApp(mavenHome) {
    sh "${mavenHome}/bin/mvn clean package"
}

node {
    def mavenHome = tool name: 'maven-3.9.6'

    stage('Checkout') {
        git 'https://github.com/example/repo.git'
    }

    stage('Build') {
        buildApp(mavenHome)
    }
}
```

---

## 12.4 Better error handling

In Scripted Pipeline we can use `try-catch-finally`.

Example:

```groovy
node {
    try {
        stage('Build') {
            sh 'mvn clean package'
        }

        stage('Deploy') {
            sh './deploy.sh'
        }
    } catch (Exception e) {
        echo "Pipeline failed: ${e}"
        currentBuild.result = 'FAILURE'
    } finally {
        echo "Cleanup tasks"
    }
}
```

This is much better than Freestyle job failure visibility.

---

## 12.5 Conditional deployments

Example:

```groovy
node {
    stage('Checkout') {
        git branch: 'dev', url: 'https://github.com/example/repo.git'
    }

    if (env.BRANCH_NAME == 'dev') {
        stage('Deploy to Dev') {
            sh './deploy-dev.sh'
        }
    } else if (env.BRANCH_NAME == 'main') {
        stage('Deploy to Prod') {
            sh './deploy-prod.sh'
        }
    }
}
```

This kind of flexible branching is a major benefit.

# 13. Different Ways to Write Scripted Pipelines

Now let us discuss different approaches.

---

## 13.1 Simple linear pipeline

Good for beginner learning and simple projects.

```groovy
node {
    stage('Checkout') {
        git 'https://github.com/example/repo.git'
    }
    stage('Build') {
        sh 'mvn clean package'
    }
    stage('Deploy') {
        sh './deploy.sh'
    }
}
```

Use case:
Small project
Simple CI/CD
One environment

---

## 13.2 Pipeline with tool integration

```groovy
node {
    def javaHome = tool name: 'jdk-17'
    def mavenHome = tool name: 'maven-3.9.6'

    env.JAVA_HOME = "${javaHome}"
    env.PATH = "${javaHome}/bin:${mavenHome}/bin:${env.PATH}"

    stage('Checkout') {
        git 'https://github.com/example/repo.git'
    }

    stage('Build') {
        sh 'mvn clean package'
    }
}
```

Use case:
When Jenkins tools are centrally configured

---

## 13.3 Pipeline with try-catch

```groovy
node {
    try {
        stage('Checkout') {
            git 'https://github.com/example/repo.git'
        }

        stage('Build') {
            sh 'mvn clean package'
        }
    } catch (err) {
        echo "Build failed"
        throw err
    }
}
```

Use case:
Error handling
Notifications
Cleanup activities

---

## 13.4 Pipeline with functions

```groovy
def checkoutCode() {
    git branch: 'main', url: 'https://github.com/example/repo.git'
}

def buildApp() {
    sh 'mvn clean package'
}

node {
    stage('Checkout') {
        checkoutCode()
    }

    stage('Build') {
        buildApp()
    }
}
```

Use case:
Reusable code
Cleaner pipeline
Same logic across stages

---

## 13.5 Pipeline with parallel execution

```groovy
node {
    stage('Checkout') {
        git 'https://github.com/example/repo.git'
    }

    stage('Parallel Testing') {
        parallel(
            "Unit Tests": {
                sh 'mvn test'
            },
            "Code Scan": {
                sh 'mvn sonar:sonar'
            }
        )
    }
}
```

Use case:
Speed improvement
Run multiple validations simultaneously

---

## 13.6 Pipeline with manual approval

```groovy
node {
    stage('Build') {
        sh 'mvn clean package'
    }

    stage('Deploy to Dev') {
        sh './deploy-dev.sh'
    }

    stage('Approval') {
        input message: 'Do you want to deploy to QA?'
    }

    stage('Deploy to QA') {
        sh './deploy-qa.sh'
    }
}
```

Use case:
Controlled promotions
Release approval flow

---

## 13.7 Pipeline with environment-specific logic

```groovy
node {
    def ENVIRONMENT = "dev"

    stage('Build') {
        sh 'mvn clean package'
    }

    if (ENVIRONMENT == "dev") {
        stage('Deploy to Dev') {
            sh './deploy-dev.sh'
        }
    }

    if (ENVIRONMENT == "qa") {
        stage('Deploy to QA') {
            sh './deploy-qa.sh'
        }
    }
}
```

Use case:
Same pipeline for multiple environments

# 14. Important Scripted Pipeline Concepts

## 14.1 tool

Used to access tools configured in Jenkins global tool configuration.

Example:

```groovy
def mavenHome = tool name: 'maven-3.9.6'
```

Why used:
No need to hardcode Maven path
Better portability

---

## 14.2 sh

Used to execute shell commands on Linux agents.

Example:

```groovy
sh 'pwd'
sh 'ls -lrt'
sh 'mvn clean package'
```

---

## 14.3 bat

Used to execute commands on Windows agents.

Example:

```groovy
bat 'dir'
bat 'mvn clean package'
```

---

## 14.4 echo

Used to print logs in build console.

Example:

```groovy
echo "Build started"
```

---

## 14.5 env

Used to access environment variables.

Example:

```groovy
echo "${env.BUILD_NUMBER}"
echo "${env.JOB_NAME}"
echo "${env.WORKSPACE}"
```

---

## 14.6 currentBuild

Used to access current build details.

Example:

```groovy
echo "Build result: ${currentBuild.currentResult}"
```

Can also set status:

```groovy
currentBuild.result = 'FAILURE'
```

---

## 14.7 withCredentials

Used to securely access usernames, passwords, tokens, keys.

Example:

```groovy
withCredentials([usernamePassword(credentialsId: 'nexus-creds', usernameVariable: 'USER', passwordVariable: 'PASS')]) {
    sh 'curl -u $USER:$PASS http://nexus-url'
}
```

Why important:
Never hardcode passwords in pipeline

# 15. Real-Time Scripted Pipeline Full Example

Below is a stronger real-time example.

```groovy
node {
    def mavenHome = tool name: 'maven-3.9.6'

    try {
        stage('Checkout') {
            git branch: 'dev', url: 'https://github.com/kkdevopsb7/maven-webapplication-project-kkfunda.git'
        }

        stage('Workspace Info') {
            sh 'pwd'
            sh 'ls -lrt'
        }

        stage('Compile') {
            sh "${mavenHome}/bin/mvn compile"
        }

        stage('Unit Test') {
            sh "${mavenHome}/bin/mvn test"
        }

        stage('Package') {
            sh "${mavenHome}/bin/mvn clean package"
        }

        stage('SonarQube Analysis') {
            sh "${mavenHome}/bin/mvn sonar:sonar"
        }

        stage('Archive') {
            archiveArtifacts artifacts: 'target/*.war', fingerprint: true
        }

        stage('Deploy to Nexus') {
            sh "${mavenHome}/bin/mvn clean deploy"
        }

        stage('Deploy to Tomcat') {
            withCredentials([usernamePassword(credentialsId: 'tomcat-creds', usernameVariable: 'TOMCAT_USER', passwordVariable: 'TOMCAT_PASS')]) {
                sh '''
                    curl -u $TOMCAT_USER:$TOMCAT_PASS \
                    --upload-file target/maven-web-application.war \
                    http://TOMCAT-SERVER:8080/manager/text/deploy?path=/mavenapp&update=true
                '''
            }
        }

        stage('Validation') {
            sh 'curl -I http://TOMCAT-SERVER:8080/mavenapp'
        }

        currentBuild.result = 'SUCCESS'
    } catch (Exception e) {
        currentBuild.result = 'FAILURE'
        echo "Pipeline failed due to: ${e}"
        throw e
    } finally {
        echo "Pipeline execution completed"
    }
}
```

# 18. Common Real-Time Problems in Scripted Pipelines

## 18.1 Wrong quotes inside shell command

Wrong:

```groovy
sh 'echo 'Hello World''
```

This is invalid because single quotes are nested incorrectly.

Correct ways:

```groovy
sh "echo 'Hello World'"
```

or

```groovy
sh 'echo Hello World'
```

or

```groovy
sh """
echo "Hello World"
"""
```

---

## 18.2 Tool not found

Problem:
`mvn: command not found`

Reason:
Tool not configured properly in Jenkins
Wrong tool name used

Fix:
Check Global Tool Configuration
Verify exact configured Maven name
Use `tool name: 'maven-3.9.6'`

---

## 18.3 Permission denied in shell script

Problem:
`./deploy.sh: Permission denied`

Fix:

```groovy
sh 'chmod +x deploy.sh'
sh './deploy.sh'
```

---

## 18.4 Git checkout issue

Problem:
Authentication failure while cloning private repo

Fix:
Use Jenkins credentials
Attach credentials in git step or scm config

---

## 18.5 Workspace path issues

Problem:
Artifact not found

Reason:
Wrong path used in deploy step

Fix:
Use workspace check:

```groovy
sh 'pwd'
sh 'ls -lrt'
sh 'ls -lrt target'
```

---

## 18.6 SonarQube step failing

Possible reasons:
Sonar server not reachable
Token missing
Project configuration missing
Scanner configuration missing

Troubleshooting:
Check network connectivity
Check credentials
Check sonar properties
Check Jenkins plugin setup

---

## 18.7 Nexus deployment failure

Possible reasons:
Wrong credentials
Wrong repository URL
pom.xml distributionManagement missing
Permission issue in Nexus

---

## 18.8 Tomcat deployment failure

Possible reasons:
Manager app not enabled
User role missing in Tomcat
Wrong URL
WAR file missing
Tomcat server down

# 19. Scripted Pipeline Best Practices

## 19.1 Keep pipeline code in Git

Never depend only on UI-created pipeline.

---

## 19.2 Do not hardcode passwords

Use Jenkins credentials.

---

## 19.3 Use stages properly

Create meaningful stage names:
Checkout
Build
Test
Scan
Package
Deploy
Validate

---

## 19.4 Add logs for troubleshooting

Use `echo`, `pwd`, `ls -lrt` wherever useful.

---

## 19.5 Use try-catch

This helps in proper failure handling.

---

## 19.6 Build once, deploy many

Do not rebuild artifact for every environment.
Build once and promote same artifact.

---

## 19.7 Use archiveArtifacts or external repositories

Store artifacts properly in Jenkins, Nexus, or Artifactory.

---

## 19.8 Separate CI and CD if needed

In large projects:
One pipeline for build and artifact publishing
Another pipeline for deployment

---

## 19.9 Keep reusable logic in shared methods or shared libraries

This reduces duplicate code.

| Configuration            | UI based                 | Code based        |

# 22. Sample Interview Answer

## Question:

Why did your team move from Freestyle jobs to Jenkins Pipelines?

## Answer:

In our project, Freestyle jobs worked initially for small automation tasks, but when CI/CD became complex, they created many operational issues. Each job had manual UI-based configuration, and maintaining multiple jobs for build, test, scan, artifact upload, and deployment became difficult. There was no proper version control for job configuration, and changes made in Jenkins UI were hard to audit. To solve this, we moved to Jenkins Pipelines. In pipelines, we wrote the full build and deployment logic as code inside a Jenkinsfile. This gave us version control, reusability, better troubleshooting, conditional execution, and easier integration with tools like Git, Maven, SonarQube, Nexus, Docker, and Tomcat. In some advanced cases, we used Scripted Pipeline because it gave us more flexibility to handle dynamic logic and real-time requirements.

# 25. Advanced Scripted Pipeline Example with Parallel and Approval
